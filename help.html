Giới thiệu

Plugin làm được gì?

Khi xây dựng design system, một trong những việc tốn thời gian nhất là duy trì typography nhất quán và responsive. Mỗi khi thay đổi token (ví dụ tăng font size Body lên 1px), bạn phải cập nhật thủ công hàng chục text styles, rồi gán lại cho hàng trăm text node — và làm lại 3 lần cho 3 breakpoint.

Typography DS giải quyết toàn bộ việc đó bằng cách kết hợp hai tính năng của Figma:

📐

Figma Variables

Lưu giá trị fontSize theo từng breakpoint. Khi chuyển mode 1280→1920, toàn bộ font size tự động scale đúng theo thiết kế.

🎨

Text Styles

Đặt tên, phân nhóm và lock toàn bộ thuộc tính (size, weight, line-height, letter-spacing) cho designer dùng từ bảng Styles.

Chuỗi liên kết: Token → Variable → Text Style → Text Node

Hệ thống Tokens

Thay vì dùng giá trị pixel cụ thể (ví dụ fontSize: 14px), typography token là tên ngữ nghĩa đại diện cho một cấp độ trong thang chữ. Plugin dùng 7 tokens từ nhỏ đến lớn:

Tại sao dùng Line Height theo px?

Line height lưu theo px tuyệt đối trong biến số Figma (để scale theo breakpoint), nhưng khi gán vào text style, plugin chuyển về % — ví dụ base tại 1280: LH 20px / Size 13px ≈ 154%. Giá trị % này tự scale tỉ lệ khi font size thay đổi, không cần biến số riêng.

Tại sao Letter Spacing không có biến số riêng?

LS cũng lưu theo % trong text style (ví dụ 8% cho Overline). Cách này tự scale theo size mà không cần thêm variable, giữ collection gọn và dễ quản lý.

Cơ chế hoạt động trong Figma

Plugin thực hiện 3 bước để tạo typography responsive:

1

Tạo Variable Collection "Typography"

Collection có 3 Mode: 1280 / 1440 / 1920. Mỗi token như size/2xl được tạo thành 1 FLOAT variable với 3 giá trị — một cho mỗi mode.

2

Tạo Text Styles, bind fontSize vào Variable

Text style Display có fontSize được bind vào variable size/2xl. Nhờ vậy, khi frame chuyển sang mode 1920, text style tự động dùng giá trị 28px thay vì 22px.

3

Đặt Variable Mode cho frame

Plugin gọi setExplicitVariableModeForCollection trên frame, đặt frame về mode 1280. Khi nhân bản frame cho breakpoint 1440/1920, clone được đặt sang mode tương ứng.

✅

Với cách này, designer chỉ cần chọn "FAN Font/Display" từ bảng Styles — font size sẽ tự đúng cho dù đang ở frame nào. Không cần nhớ giá trị pixel, không bao giờ bị sai breakpoint.

Tab Tokens

✏️

Chỉnh sửa token values

Bảng edit inline: Font Size, Line Height (px), Letter Spacing (px) — mỗi bảng 7 token × 3 breakpoint. Bấm Apply để áp dụng ngay vào Figma.

🗂

CRUD style definitions

Xem, sửa, thêm, xoá 46 text style definitions ngay trong plugin. Chia thành 10 nhóm (Page, Body, Form, Table, Button...). Mỗi style có đủ: name, token, weight, lh%, ls%, textCase, mono.

💾

Lưu trữ trong file Figma

Dữ liệu token và style được lưu bằng figma.root.setPluginData — nằm trong file Figma, tất cả thành viên trong team đều đọc/ghi được khi mở file.

🔁

Reset về mặc định

Khôi phục cả bảng token lẫn toàn bộ style definitions về giá trị hardcoded gốc. Hữu ích khi thử nghiệm xong muốn quay về baseline.

🟠

Unsaved indicator

Nút Apply ngay chuyển cam và hiện dấu ● khi có thay đổi chưa được áp dụng. Tự xoá khi Apply hoặc Reset. Tránh quên apply sau khi chỉnh token.

📤

Export / Import JSON

Export toàn bộ token + style definitions ra file typography-tokens.json. Import ngược lại vào bất kỳ file Figma nào để đồng bộ cài đặt giữa các dự án — không cần Reset + nhập tay.

Hướng dẫn

Cách sử dụng

Lần đầu sử dụng

1

Load plugin vào Figma

Mở Figma → Plugins → Development → Import plugin from manifest → chọn file plugin/manifest.json trong thư mục repo.

2

Nhập tên nhóm styles

Ô Nhóm ở đầu plugin — mặc định là FAN Font. Tên này là prefix cho tất cả styles, ví dụ FAN Font/Body/Default. Thay đổi tuỳ dự án.

3

Chọn frame và bấm Chuẩn hoá

Chọn một hoặc nhiều frame từ danh sách (tick checkbox), bấm Chuẩn hoá. Plugin sẽ: tạo variables → tạo 46 text styles → gán styles cho tất cả text node trong frame.

4

(Tuỳ chọn) Nhân bản frame

Tích Nhân bản frame theo breakpoint, chọn 1440 / 1920. Plugin clone frame, resize và đặt Variable Mode đúng breakpoint. Mỗi frame sẽ hiển thị font size đúng màn hình của nó.

Thay đổi giá trị token

1

Mở Tab Tokens → Font Size

Sửa các ô trong bảng Font Size / Line Height / Letter Spacing. Ví dụ tăng base tại 1440 từ 14px lên 15px.

2

Bấm Apply ngay

Plugin lưu data vào file Figma, cập nhật tất cả Figma Variables và tạo lại 46 text styles với giá trị mới. Các frame đang dùng styles đó sẽ tự cập nhật.

Thêm hoặc xoá một text style

1

Mở Tab Tokens → Styles → mở nhóm tương ứng

Ví dụ muốn thêm style mới vào nhóm Body, mở sub-accordion Body.

2

Bấm "+ Thêm style" hoặc nhấn ✕ để xoá

Điền tên, chọn token, weight, lh%, ls%. Nút M bật monospace font (Courier New). Nút Aa/AA bật uppercase text case.

3

Bấm Apply ngay

Style mới được tạo trong Figma. Style bị xoá sẽ bị remove khỏi Local Styles (chú ý: text node nào đang dùng style đó sẽ mất style assignment).

⚠️

Lưu ý: Khi xoá một style và bấm Apply, style đó bị xoá khỏi Figma. Các text node đang dùng style đó sẽ cần gán lại thủ công sau khi chạy Chuẩn hoá.

Cấu trúc file được export

File gồm 2 phần: CSS custom properties (theo breakpoint) và các class cho từng text style.

/* ── Phần 1: Token variables ── */
:root {
  /* font-size */
  --size-xs: 10px;   --size-sm: 11px;   --size-base: 13px;
  --size-md: 14px;  --size-lg: 16px;  --size-xl: 18px;
  --size-2xl: 22px;
  /* line-height */
  --lh-base: 20px; --lh-xl: 28px; /* ... */
  /* letter-spacing */
  --ls-0: 0px;  --ls-tight: -0.2px;  --ls-tighter: -0.4px;
}

@media (min-width: 1440px) {
  :root { --size-sm: 12px; --size-base: 14px; /* ... */ }
}
@media (min-width: 1920px) {
  :root { --size-sm: 13px; --size-base: 16px; /* ... */ }
}

/* ── Phần 2: Text style classes ── */
.text-display {
  font-family: Arial, sans-serif;
  font-size: var(--size-2xl);
  font-weight: 600;
  line-height: 130%;
}
.text-body-default {
  font-size: var(--size-base);
  font-weight: 400;
  line-height: 170%;
}

Hỏi & Đáp

Câu hỏi thường gặp

Tại sao một số text node bị "bỏ qua" khi Chuẩn hoá? +

Plugin bỏ qua text node khi gặp 4 trường hợp sau — và báo cáo chi tiết lý do ngay dưới nút Chuẩn hoá:

  • font lạ — node không dùng Arial (font chính của dự án). Tất cả font khác được gom vào nhóm này.
  • mixed font — node có nhiều font khác nhau trong cùng 1 text box (highlight từng phần).
  • 14px/Regular (ví dụ) — font size không khớp bất kỳ token nào ở bất kỳ breakpoint nào. Báo cáo hiển thị dạng Npx/Weight để dễ tra.
  • style chưa tạo — style chưa tồn tại trong file (thường do đổi tên nhóm Nhóm mà chưa chạy lại Chuẩn hoá).

Khi chọn nhiều frame, mỗi frame báo riêng số gán thành công và top 3 lý do bỏ qua.

Font Family có thể thay đổi trong plugin không? +

Hiện tại font family (Arial) được hardcode trong plugin/code.js. Để đổi font, mở file và sửa 2 biến ở đầu file:

var TARGET_FONT = 'Arial'; // font chính
var FONT_MONO = 'Courier New'; // font cho style mono
var WEIGHT_MAP = { 400: 'Regular', 500: 'Bold', 600: 'Bold' };

Arial không có Medium/SemiBold nên weight 500 và 600 đều map về Bold. Nếu dùng font khác có đủ weight, cập nhật WEIGHT_MAP theo.

Cả team có dùng chung cài đặt token được không? +

Có. Dữ liệu token và style definitions được lưu bằng figma.root.setPluginData — dữ liệu nằm trong bản thân file Figma (.fig), không phải trên máy cá nhân. Mọi người mở cùng file đều đọc được cùng một bộ token.

Khi một thành viên thay đổi token và bấm Apply, thay đổi được lưu vào file. Thành viên khác chỉ cần mở lại plugin là thấy bộ token mới nhất.

Tại sao có 3 style giống nhau không phân biệt được (Body/Secondary, Label/Disabled, Input/Placeholder)? +

Ba styles này có cùng font size, weight và line height với variant Default của chúng — chúng chỉ khác màu chữ, mà màu sắc không thể tự động detect từ typography token.

Plugin sẽ gán chúng về variant Default (ví dụ Body/Default thay vì Body/Secondary). Bạn cần gán thủ công cho những node cần dùng Secondary/Disabled/Placeholder.

Bấm Chuẩn hoá nhiều lần có bị ghi đè hay tạo duplicate không? +

Không. Tất cả thao tác đều idempotent:

  • Variable Collection "Typography" đã tồn tại → tái sử dụng, cập nhật giá trị.
  • Text style đã tồn tại cùng tên → cập nhật, không tạo mới.
  • Text node đã có style → override sang style mới (nếu khớp).

An toàn để chạy lại bất kỳ lúc nào sau khi thay đổi token hoặc thêm text node mới vào frame.

Nhân bản frame tạo ra frame ở đâu? Tên frame clone là gì? +

Frame clone được đặt ngay bên phải frame gốc (cách nhau 80px), cùng Y position. Tên frame là tên frame gốc + hậu tố breakpoint: ví dụ frame gốc Màn hình chính → clone Màn hình chính-Z-1440 và Màn hình chính-Z-1920.

Nếu đã có clone tên trùng từ lần chạy trước, plugin xoá và tạo lại để tránh duplicate. Frame gốc không bị ảnh hưởng — và được đặt mode 1280 tự động.

Tính năng nhân bản áp dụng cho tất cả frame được chọn trong danh sách — không chỉ frame đầu tiên.

Muốn reset về giá trị gốc sau khi đã chỉnh token thì làm sao? +

Tab Tokens → nút "Reset về default" ở footer. Thao tác này:

  • Xoá dữ liệu đã lưu trong file Figma.
  • Khôi phục toàn bộ bảng Font Size, Line Height, Letter Spacing về giá trị hardcoded.
  • Khôi phục toàn bộ 46 style definitions về mặc định.
  • Cập nhật giao diện ngay lập tức.

Bạn cần bấm Apply ngay thêm một lần nữa để áp dụng giá trị gốc vào Figma Variables và Text Styles.

Nút Apply ngay chuyển cam là sao? +

Đó là unsaved indicator — báo hiệu bạn đã thay đổi giá trị trong Tab Tokens nhưng chưa bấm Apply. Nút cũng hiện thêm dấu ● để dễ nhận ra.

Indicator tự tắt khi:

  • Bấm Apply ngay — thay đổi được ghi vào Figma.
  • Bấm Reset về default — data bị rollback.
  • Plugin load lại data từ Figma (mở plugin lần đầu hoặc sau Reset).
Làm sao đồng bộ token sang file Figma khác? +

Dùng cặp nút Export JSON / Import JSON ở footer Tab Tokens:

  1. Mở File A (có token cần sao chép) → bấm Export JSON → tải về typography-tokens.json.
  2. Mở File B (file đích) → bấm Import JSON → chọn file vừa tải.
  3. Kiểm tra bảng token vừa được điền vào, nút Apply chuyển cam.
  4. Bấm Apply ngay để ghi vào Figma Variables và Text Styles của File B.

File JSON chứa đồng thời tokenData (size/LH/LS) và styleData (46 style definitions).

"Tạo trang Demo" tạo ra gì? Có ghi đè lần chạy trước không? +

Plugin tạo 3 frame Demo 1280, Demo 1440, Demo 1920 side by side trên page hiện tại — mỗi frame là showcase đầy đủ 46 text roles trong 9 sections (Page, Body, Form, Table, Button, Badge, Nav, Notif, Approval).

Mỗi text node được gán textStyleId từ local styles, mỗi frame bind đúng Variable Mode — chuyển mode sẽ thấy typography scale theo breakpoint.

Nếu 3 frame Demo đã tồn tại, plugin xoá và tạo lại. Tuy nhiên, nội dung text bạn đã chỉnh sửa trong Demo 1280 sẽ được giữ nguyên — plugin đọc text đó trước khi build lại. Để reset về text mặc định, xoá frame Demo 1280 trước khi chạy.

Node.js scripts (read-file.js, apply-styles.js) dùng để làm gì? +

Đây là các script phụ trợ chạy ngoài Figma qua REST API, chỉ dùng để phân tích và báo cáo:

  • node read-file.js — liệt kê tất cả text node, font size, styles hiện có trong file.
  • node apply-styles.js — mô phỏng quá trình match style, báo cáo node nào sẽ được gán style gì (read-only, không thay đổi Figma).

Figma REST API không hỗ trợ PATCH để gán textStyleId cho node — việc gán styles thực sự chỉ làm được qua plugin. Cần tạo file .env với FIGMA_TOKEN và FIGMA_FILE_KEY.

Góc nhìn Developer — Cơ chế kỹ thuật

Vòng lặp triệt tiêu zoom

Typography Demo dùng một kỹ thuật gọi là scaled viewport: toàn bộ nội dung được bọc trong một div#dm-scaler và thu phóng bằng CSS transform: scale(X), trong đó X được tính theo công thức:

scale = outerContainerWidth / viewportWidth
// Ví dụ: container 900px, viewport 1440px → scale = 0.625
scaler.style.transform = 'scale(0.625)';

Mục tiêu: hiển thị đúng 1 trang 1440px vừa khít trong 900px màn hình thực tế — y chang cách Figma render canvas của nó.

Bây giờ, khi bạn nhấn Ctrl +, trình duyệt Chrome/Firefox làm điều này:

  1. Zoom toàn bộ trang lên (ví dụ 125%)
  2. Hệ quả: window.innerWidth thu nhỏ lại (vì mỗi CSS pixel nay chiếm nhiều điểm ảnh màn hình hơn) — ví dụ từ 900px → 720px
  3. Trình duyệt kích hoạt sự kiện resize
  4. Demo đang lắng nghe: window.addEventListener('resize', _dmResize)
  5. _dmResize chạy lại dmSetVP(), tính lại scale: 720 / 1440 = 0.5 (nhỏ hơn trước)
  6. Transform scale giảm → triệt tiêu đúng bằng lượng zoom đã tăng

Kết quả cuối cùng: font size nhìn thấy trên màn hình không đổi. Không phải vì ai cố tình khoá zoom — mà vì cơ chế responsive scaling tự động chống lại zoom như một phản xạ không chủ đích.

🔍

Đây không phải user-scalable=no hay bất kỳ lệnh "khoá zoom" nào. Chỉ là resize listener đang làm đúng nhiệm vụ của nó — và vô tình tạo ra hiệu ứng phụ này.

Lịch sử: từ px → rem → clamp() — Ngành đã đi con đường nào?

Giai đoạn 1 — Pixel tuyệt đối (2000–2012)

Tất cả font size đều dùng px. Ctrl+ zoom hoạt động ở IE6 bằng cách phóng to toàn trang (layout zoom). Không ai nghĩ nhiều đến accessibility.

Giai đoạn 2 — em và rem (2012–2018)

Cộng đồng phát hiện ra: nếu dùng rem thay vì px, font size tự động scale theo cài đặt font của trình duyệt (thường là 16px baseline). Người dùng có thể vào Settings → Font Size → tăng lên và toàn bộ web tự điều chỉnh — không cần Ctrl+.

/* Cách cũ — px cứng, không scale với user preference */
body { font-size: 16px; }
h1   { font-size: 32px; }

/* Cách đúng — rem scale theo browser base font */
:root { font-size: 100%; }   /* = 16px theo browser default */
body  { font-size: 1rem; }   /* 16px */
h1    { font-size: 2rem; }   /* 32px, nhưng scale nếu user tăng base */

Giai đoạn 3 — Fluid typography với clamp() (2020–nay)

Đây là tiêu chuẩn hiện đại nhất. Thay vì step-breakpoints ("tại 1440px thì dùng 14px"), font size trượt mượt theo viewport width:

/* Font size tự chạy từ 13px (tại 1280px) đến 16px (tại 1920px) — không cần @media */
body {
  font-size: clamp(0.8125rem, 0.489rem + 0.508vw, 1rem);
  /*          min         fluid           max        */
}

Ưu điểm: không có "điểm gãy", không cần breakpoint cứng, scale mượt hoàn toàn. Nhược điểm: khó kiểm soát và debug hơn, không phù hợp với design system dùng fixed token scale.

/* Hoặc đơn giản hơn: dùng rem native từ đầu trong production */

Đây là quyết định của developer, không phải của plugin — plugin đúng khi export px vì đó là giá trị design token gốc.

Phần 1 — Bức tranh workflow của designer Figma hiện đại (2024–2026)

Họ vẫn tạo frame riêng cho từng breakpoint — nhưng ít hơn bạn nghĩ

Một misconception phổ biến: "designer giỏi chỉ cần design 1 màn hình rồi dùng Auto Layout là responsive hết." Thực tế không phải vậy. Khảo sát các design team lớn (Atlassian, Shopify, Airbnb, các agency Đông Nam Á) cho thấy họ đều tạo ít nhất 2 và thường là 3 breakpoint frames riêng biệt cho mỗi màn hình quan trọng.

Tại sao? Vì nhiều quyết định layout thực sự không thể tự động hoá: sidebar ẩn thành hamburger ở mobile, 3 cột thành 1 cột, navigation top thành bottom tab bar. Những thứ đó đòi hỏi designer phải vẽ riêng — không có công cụ nào tự làm được.

Điều họ làm khác bạn: lớp Component Library bên dưới

1

Màu sắc, typography, spacing, border radius — tất cả được định nghĩa là Variables trước khi vẽ bất cứ thứ gì. Đây là tầng "nguyên liệu thô". Plugin của bạn đã làm điều này cho typography.

Button, Input, Badge, Card, Table row... mỗi thứ là 1 component Figma với Auto Layout. Component không biết nó sẽ xuất hiện ở breakpoint nào — nó chỉ biết "tôi rộng bao nhiêu thì tôi trông như thế này".

3

Frames 1280/1440/1920 — dùng component từ bước 2, sắp xếp theo layout của từng breakpoint. Đây là bước bạn đang làm.

4

Ghép template + component + nội dung thật. Prototype click-through để test flow.

5

Developer mở Figma Dev Mode, inspect từng element, xem CSS snippet được tự generate. Không cần Zeplin, không cần InVision nữa.

Phần 3 — Figma Dev Mode: handoff không còn là nỗi đau

Đây là thứ đã thay đổi hoàn toàn quy trình designer-developer từ 2023 đến nay. Nếu bạn còn đang export PNG, đo pixel bằng tay, hay dùng Zeplin — bạn đang làm theo cách 2018.

Dev Mode là gì?

Figma Dev Mode (Shift+D trong Figma) là chế độ xem dành cho developer, trong đó:

  • Click vào bất kỳ element nào → xem CSS snippet được generate tự động
  • Inspect spacing, color, typography — hiển thị theo đúng design token nếu đã dùng Variables
  • Download assets (icon, image) ngay tại chỗ
  • So sánh "implemented vs design" — dev có thể overlay code lên design để check
  • Comment và annotation ngay trong file

CSS được generate trông như thế nào?

/* Designer click vào Button trong Figma Dev Mode, thấy: */
/* Auto-generated by Figma Dev Mode */
.button-primary {
  display: flex;
  align-items: center;
  justify-content: center;
  gap: 8px;
  padding: 10px 20px;
  background: #2D6BE4;          /* nếu dùng Variable → hiển thị tên token */
  border-radius: 6px;
  font-size: 15px;               /* nếu dùng Text Style → hiển thị style name */
  font-weight: 500;
  color: #FFFFFF;
  width: fit-content;            /* từ Hug width */
}

Quan trọng: nếu bạn đã dùng Variables và Text Styles đúng cách (như plugin của bạn tạo ra), CSS snippet sẽ hiển thị tên token thay vì giá trị hardcode — developer thấy --size-base thay vì 13px. Đây là điểm mạnh cực lớn của workflow Variables-first.

Phần 4 — CSS thực tế: developer làm gì với file thiết kế của bạn?

Đây là phần bạn hỏi trực tiếp: ngoài việc "convert sang rem", thực tế mọi người làm như thế nào?

Con đường 1 — CSS Custom Properties + Media Query (phổ biến nhất)

Đây là cách đại đa số team đang làm, và cũng là cách plugin của bạn export ra. Developer nhận file tokens.css từ bạn và dùng như sau:

/* tokens.css từ plugin export — dev KHÔNG sửa file này */
:root {
  --size-base: 13px;
  --size-xl:   18px;
}
@media (min-width: 1440px) {
  :root { --size-base: 14px; --size-xl: 20px; }
}

/* component.css — dev viết, dùng token */
body    { font-size: var(--size-base); }
h1      { font-size: var(--size-xl); }
.label  { font-size: var(--size-sm); font-weight: 500; }

/* Kết quả: tự động đúng ở mọi breakpoint — không cần viết @media lại */

Cách này đơn giản, không cần tool thêm, và đây là điều plugin của bạn đã tối ưu hoàn toàn. Developer chỉ cần import một file là xong responsive typography.

Con đường 2 — Tailwind CSS (phổ biến trong startup và web modern)

Tailwind là framework utility-first — thay vì viết CSS, developer viết class trực tiếp vào HTML:

<!-- Không dùng Tailwind -->
<h1 class="page-title">Tờ trình</h1>

<!-- Dùng Tailwind -->
<h1 class="text-[22px] lg:text-[24px] xl:text-[28px] font-bold leading-[1.3]">
  Tờ trình
</h1>

Responsive prefix trong Tailwind: lg: = min-width 1024px, xl: = 1280px, 2xl: = 1536px. Developer viết 3 giá trị một lần, tự responsive.

Thực tế với SharePoint: SharePoint dùng SPFx (SharePoint Framework) — về cơ bản là React. Tailwind hoàn toàn dùng được trong SPFx, nhưng nhiều team enterprise vẫn dùng CSS Modules hoặc Fluent UI (Microsoft's design system). Nếu dev của bạn đang dùng SPFx + Fluent UI, tokens.css vẫn work tốt.

Con đường 3 — rem thuần túy (cách đúng về accessibility)

Đây là câu bạn hỏi: "Ngoài 'convert sang rem', thực tế mọi người làm như thế nào?"

Thật ra convert px → rem đơn giản hơn bạn nghĩ. Có 3 cách:

/* Cách 1: Viết tay — chia cho 16 */
body { font-size: 0.8125rem; }  /* 13 ÷ 16 = 0.8125 */

/* Cách 2: CSS calc() — giữ px trong token, tính rem trong component */
body { font-size: calc(var(--size-base) / 16 * 1rem); }

/* Cách 3: Đặt root font-size = 62.5% — trick nổi tiếng */
:root { font-size: 62.5%; }  /* 16px × 62.5% = 10px */
body  { font-size: 1.3rem; }  /* 1.3 × 10px = 13px — dễ tính hơn */
h1    { font-size: 2.2rem; }  /* 2.2 × 10px = 22px */

Cách 3 (62.5% trick) từng rất phổ biến nhưng hiện nay có tranh cãi vì gây confusion. Cách 2 (calc với token) là cách sạch nhất và tương thích tốt nhất với workflow token-based như của bạn.

Thực tế có bao nhiêu team convert sang rem? Thẳng thắn mà nói: khoảng 40–50% team enterprise không bao giờ convert — họ dùng px trong production và biết rõ đó là technical debt accessibility. Với internal tool dùng trong môi trường corporate (màn hình cố định, người dùng ít khi thay đổi font size hệ thống), họ chấp nhận trade-off này. Đây là thực tế, không phải lý tưởng.

Con đường 4 — clamp() fluid typography (cutting edge, chưa phổ biến)

/* font-size tăng dần từ 13px (tại 1280px) đến 16px (tại 1920px) — không cần @media */
body {
  font-size: clamp(
    0.8125rem,           /* min: 13px */
    0.489rem + 0.508vw,  /* fluid: tính bằng công thức */
    1rem                 /* max: 16px */
  );
}

Đẹp về lý thuyết, nhưng có 2 vấn đề thực tế: (1) designer không design "fluid" trong Figma — họ vẫn dùng breakpoint frames, tạo ra conflict giữa design và code; (2) khó debug và khó match chính xác với Figma. Hiện tại chỉ phổ biến trong marketing/landing page, chưa phổ biến trong enterprise app.

1Mang blog này sang project mới

Toàn bộ blog là một panel tự chứa trong help.html. Để tái sử dụng ở project khác:

  • Copy file help.html sang project mới (hoặc tạo file mới từ template)
  • Copy file BLOG.md vào cùng thư mục — đây là "bộ nhớ" để Claude hiểu cách thêm bài
  • Mở Claude Code trong project mới và nói:
Prompt để khởi động blog ở project mới "Đọc BLOG.md để hiểu cách thêm bài blog vào help.html, sau đó thêm bài đầu tiên về [chủ đề]"

Claude sẽ đọc BLOG.md, hiểu cấu trúc anchor BLOG_NAV_END / BLOG_ARTICLES_END, và tự thêm đúng vị trí — không cần giải thích thêm.

💡 CSS blog được đặt trong block /* ═══ Blog Panel ═══ */ trong <style> — dễ tách ra nếu muốn dùng riêng.

2Thêm bài mới ở project này

Nói với Claude Code theo cú pháp sau:

Cú pháp cơ bản "Thêm bài blog mới vào help.html, chủ đề: [X], nội dung: [Y]"

Ví dụ cụ thể:

# Bài phân tích ngắn "Thêm bài blog về roadmap plugin, nội dung: 3 phases tiếp theo với ưu tiên" # Bài có cấu trúc rõ "Thêm bài blog case study: Typography DS trong dự án thực tế, dùng compare-table và verdict callout" # Bài kỹ thuật "Thêm bài blog giải thích cơ chế auto-assign styles, dành cho developer muốn hiểu code"

Claude sẽ tự chọn ID tiếp theo (a02, a03...), chèn đúng vị trí anchor, và dùng CSS classes phù hợp.

3Mẹo để ra lệnh hiệu quả hơn

Bài viết sẽ tốt hơn khi bạn chỉ định thêm một số yếu tố:

Chỉ định giọng văn:

"... giọng chuyên gia UX/UI" ← phân tích sâu, thuật ngữ ngành "... giọng thực tế, dùng ví dụ cụ thể" ← dễ đọc, bớt hàn lâm "... dành cho developer" ← tập trung kỹ thuật, code snippets

Chỉ định cấu trúc bài:

"... có bảng so sánh" ← dùng .compare-table "... có điểm mạnh/yếu" ← dùng .score-card.pro/.con "... có rating priorities" ← dùng .rating-row "... kết thúc bằng verdict" ← dùng .verdict callout "... bài ngắn, chỉ 3 điểm chính" ← không cần đủ components

Cung cấp outline trước:

"Thêm bài blog, chủ đề: Tại sao không dùng Tokens Studio, nội dung: - Tokens Studio quá phức tạp cho team nhỏ - Setup mất 2 ngày vs Typography DS 30 phút - Không cần GitHub sync nếu chỉ 1 file Figma - Khi nào nên dùng Tokens Studio thay thế"
